1. MISSION
============================================================

Build the MCP Gateway — the central routing layer that every agent in Fedora
AI OS uses to talk to external tools. You own the core proxy: accepting JSON-
RPC requests, routing them to the correct backend, and returning responses.
You also own schema validation (rejecting malformed requests before they hit
backends) and the YAML configuration system. By month 3, you will bridge the
gateway to LangGraph so workflows can call tools through the gateway.

Your work is the single most critical deliverable for the project right now.
The CRITICAL_PATH.md file says it directly: "The next missing capability is
MCP Gateway — Core Proxy. Without the gateway, no component can talk to any
other component." You are building the foundation that everything else sits on.

3. COMPONENTS OWNED
============================================================

A. MCP GATEWAY SERVER (Month 1)
   - FastAPI/Starlette HTTP server
   - Health endpoint: GET /health → 200
   - MCP endpoint: POST /mcp — accepts JSON-RPC 2.0
   - CLI: gateway serve --port 8000
   - Files: gateway/server.py, gateway/cli.py

B. CORE PROXY ROUTING (Month 1)
   - JSON-RPC 2.0 request parser
   - Request router: tool name → backend URL resolution
   - httpx-based HTTP forwarding
   - Response passthrough to caller
   - Error handling: 502 for unreachable backend, 400 for malformed input
   - Files: gateway/router.py

C. YAML CONFIGURATION SYSTEM (Month 1)
   - Backend definitions: name, url, prefix, timeout
   - Configuration loaded at startup, validated
   - Hot-reload not required in V1
   - Files: gateway/config.py, config.yaml

D. SCHEMA VALIDATION (Month 2)
   - jsonschema validation on incoming requests
   - Per-backend JSON Schema files
   - Schema registry loaded from config
   - 400 response with field-level error details
   - Files: gateway/validation.py, gateway/schemas/

E. BACKEND INTEGRATIONS (Month 2)
   - Real backends: GitHub API, Telegram Bot API, system echo
   - Each backend has a schema, config entry, and test
   - Prove the gateway works with real services
   - Files: gateway/backends/ (backend-specific adapters)

F. LANGGRAPH BRIDGE (Month 3)
   - Wrap v3 engine steps as LangGraph nodes
   - LangGraph nodes call MCP Gateway for tool execution
   - State adapter: WorkflowState ↔ LangGraph state
   - Full round-trip: LangGraph → Gateway → Backend → Gateway → LangGraph
   - Files: implementation/phase1/langgraph_bridge/

Issue 1: Create gateway server scaffold with health endpoint
  Description: Build the FastAPI server skeleton. CLI entrypoint via

  Difficulty: Easy
  Acceptance Criteria:
    - 'gateway serve --port 8000' starts server
    - curl localhost:8000/health → 200 + JSON {'status': 'healthy'}
    - Ctrl+C shuts down cleanly

Issue 2: Implement YAML config loader for backend definitions
  Description: Read config.yaml at startup. Parse backend entries:
  name, url, prefix, timeout. Validate required fields. Return parsed
  config dict. Handle missing config file gracefully (error message).
  Difficulty: Easy
  Acceptance Criteria:
    - config.yaml with 2 backends → loader returns list of Backend objects
    - Missing required field → ValueError with field name
    - Missing config file → FileNotFoundError with helpful message
    - Test: valid config loads, invalid config raises error

Issue 3: Implement JSON-RPC request parser
  Description: Parse POST /mcp body. Validate JSON-RPC 2.0 format:
  jsonrpc='2.0', method present, id present. Extract method and params.
  Return 400 on any validation failure with descriptive error.
  Difficulty: Medium
  Acceptance Criteria:
    - Valid JSON-RPC → parsed into Python dict/object
    - Missing jsonrpc field → 400 'missing jsonrpc version'
    - Missing method → 400 'missing method'
    - Non-JSON body → 400 'invalid JSON'
    - Test: 4 test cases (valid, missing field × 2, invalid JSON)

Issue 4: Implement backend routing — tool name to URL
  Description: Given parsed JSON-RPC, resolve method to backend URL.
  Match by exact tool name or by server prefix. Forward request via
  httpx. Return backend response to caller as-is.
  Difficulty: Medium
  Acceptance Criteria:
    - Tool 'echo' → resolved to echo backend URL from config
    - Tool 'github.list_repos' → resolved to github backend URL
    - Unknown tool → 404 'unknown tool'
    - Response from backend returned unchanged to caller
    - Test: mock backend, round-trip

Issue 5: Implement error handling — 502 and 400
  Description: Backend unreachable → 502 with JSON error body.
  Backend timeout → 502. Invalid JSON-RPC → 400. Error body includes:
  error_code, message, detail (backend name, original error if safe).
  Difficulty: Easy
  Acceptance Criteria:
    - Backend down → 502 {'error': 'backend_unreachable', 'detail': ...}
    - Request timeout → 502 {'error': 'backend_timeout', 'detail': ...}
    - Malformed request → 400 {'error': 'invalid_request', 'detail': ...}
    - No internal tracebacks leaked to response body
    - Test: 3 error scenarios

Issue 6: Write proxy integration tests
  Description: Full integration test suite. Mock backend via httpx
  test client or simple HTTP server. Test scenarios: successful round-
  trip, backend timeout, backend 500, invalid JSON, unknown tool.
  Difficulty: Medium

ESSENTIAL (Month 1):

1. JSON-RPC 2.0 Specification
   https://www.jsonrpc.org/specification
   It's 4 pages. Read the whole thing. Understand: request object,
   response object, error object, notification (no id).

2. FastAPI Documentation — First Steps
   https://fastapi.tiangolo.com/tutorial/first-steps/
   Focus: path operations, request body, response models, HTTPException

3. httpx Documentation — Quickstart
   https://www.python-httpx.org/quickstart/
   Focus: async client, making requests, handling responses, timeouts

4. PyYAML Documentation
   https://pyyaml.org/wiki/PyYAMLDocumentation
   Focus: yaml.safe_load, nested dictionaries

RECOMMENDED (Month 2):

5. JSON Schema Specification (Draft 2020-12)
   https://json-schema.org/draft/2020-12/json-schema-core
   Focus: type, properties, required, additionalProperties

6. jsonschema Python Library
   https://python-jsonschema.readthedocs.io/
   Focus: validate(), ValidationError, schema registry

7. GitHub REST API Documentation
   https://docs.github.com/en/rest
   Focus: Repositories API, Issues API, authentication

8. Telegram Bot API
   https://core.telegram.org/bots/api
   Focus: sendMessage, getUpdates, making requests

RECOMMENDED (Month 3):

9. LangGraph Documentation — Quick Start
   https://langchain-ai.github.io/langgraph/tutorials/introduction/
   Focus: StateGraph, add_node, add_edge, compile, invoke

10. LangGraph Checkpointing
    https://langchain-ai.github.io/langgraph/how-tos/persistence/
    Focus: SqliteSaver, thread_id, checkpoint structure

11. TypedDict in Python (PEP 589)
    https://peps.python.org/pep-0589/
    Focus: class State(TypedDict), required and optional keys
