1. MISSION
============================================================

You are the gatekeeper of quality. Every line of code that ships to users
must pass through tests you designed, benchmarks you ran, and validation
you performed. Your job is to build the test infrastructure for the MCP
Gateway, write integration tests that catch bugs before they reach users,
create benchmark suites that measure performance, and provide reliable
bug reproduction steps when something breaks.

You are not writing the gateway itself — Lakshay Bharti owns that. You are
testing it. If the gateway claims to route requests correctly, you prove it
(or prove it doesn't). If Ojasvi's auth module claims to refresh tokens, you
write the test that forces token expiry and verifies the refresh.

A beginner who can write good tests is more valuable than a beginner who
can write features, because the tester makes every other team member better.

2. WHY THIS ROLE MATTERS
============================================================

The current project has a solid test foundation:

  From CURRENT_STATE.md:
  - 6 reliability tests (split-brain, compensation cascade, atomic failure,
    time jump, zombie management, backup recovery) — all passing
  - Production readiness suite: Chaos Monkey with 30-step workflow under
    random SIGKILL, all 30 artifacts intact after recovery

This testing culture was built for the workflow engine. It does NOT exist
for the MCP Gateway, which currently has ZERO tests.

The CRITICAL_PATH.md says the gateway is the next bottleneck. NEXT_ACTIONS.md
has tasks #9 (Gateway Integration Tests) and broader testing needs across
every gateway subsystem. Your work is what prevents the gateway from shipping
with broken routing, leaking tokens, or crashing under load.

Without you, the gateway goes to users untested. With you, every PR includes
tests that prove the code works. That's the difference between a prototype
and production-ready infrastructure.

4. LEARNING ROADMAP
============================================================

WEEK 1–2: pytest Fundamentals
  - Learn: pytest basics — test functions, assertions, fixtures
  - Learn: pytest markers, parametrize, skip/xfail
  - Learn: pytest-cov for coverage reports
  - Practice: Write 10 tests for a simple Python function

WEEK 3–4: HTTP Testing
  - Learn: httpx test client (ASGI transport for FastAPI)
  - Learn: How to mock HTTP responses (responses library or httpx MockTransport)
  - Learn: How to test async code with pytest-asyncio
  - Practice: Write integration tests for a FastAPI endpoint

WEEK 5–6: Mocking and Fixtures
  - Learn: unittest.mock (Mock, MagicMock, patch)
  - Learn: pytest fixtures — scope, autouse, yield for teardown
  - Learn: Factory pattern for test data
  - Practice: Create a mock OAuth server that returns controlled responses

WEEK 7–8: Benchmarking
  - Learn: time.perf_counter for high-precision timing
  - Learn: statistics module (mean, median, stdev, quantiles)
  - Learn: memory_profiler for memory tracking
  - Practice: Benchmark an HTTP endpoint with 1000 requests

WEEK 9–10: Chaos Engineering
  - Learn: Chaos engineering principles (Netflix Simian Army)
  - Learn: How to inject failures (terminate process, block port, add latency)
  - Learn: signal module for sending SIGKILL/SIGTERM
  - Practice: Write a script that randomly kills a server and verifies recovery

WEEK 11–12: Regression Testing
  - Learn: How to write a minimal reproduction case
  - Learn: git bisect for finding when a bug was introduced
  - Learn: How to organize regression tests (one file per bug)
  - Practice: Take a real bug, reproduce it, fix it, write a test
