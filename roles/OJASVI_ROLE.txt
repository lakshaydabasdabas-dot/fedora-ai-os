1. MISSION
============================================================

Make the MCP Gateway secure enough to talk to real services. Your job is to
ensure that API tokens are never in plaintext, that the gateway runs in a
locked-down Podman container with dropped capabilities, that SELinux policies
exist (even if permissive at first), and that every backend connection is
authenticated. By month 3, you will implement the first version of the
Security Agent's trust level enforcement — the gating mechanism that decides
whether an agent action runs silently, notifies the user, or requires explicit
approval.

============================================================
2. WHY THIS ROLE MATTERS
============================================================

The MCP Gateway is the central routing layer for the entire Fedora AI OS.
Every agent — CEO, Security, Desktop, Runtime — sends tool calls through this
gateway. If the gateway leaks API keys or runs with root privileges, the
entire system is compromised before anything else ships.

Right now the gateway is a scaffold with NO security. No OAuth. No token
management. No container isolation. No SELinux policy. No threat model.

You close that gap. Your work is what makes it safe for the gateway to talk to
GitHub, Telegram, Google Calendar, and every other backend the project depends
on. Without your work, the gateway cannot connect to any real service.

============================================================
3. FEDORA AI OS COMPONENTS OWNED
============================================================

You own three security layers, in priority order:

A. SECRETS MANAGEMENT
   - OAuth token storage in Linux Secret Service API (not plaintext files)
   - Token rotation: refresh before expiry, support rotation
   - Authorization header injection into proxied requests
   - Multiple backend support: each backend gets its own credential
   - Files: implementation/phase1/mcp-gateway/gateway/auth.py

B. PODMAN CONTAINER SECURITY
   - Rootless Podman for ALL services (gateway, Redis, Goose, OTel collector)
   - Dropped capabilities per container (cap_drop=ALL, cap_add minimal)
   - Read-only rootfs where possible
   - No host network mode (use podman network)
   - Security-relevant Quadlet directives verified
   - Files: implementation/phase1/containers/

C. SELINUX POLICY (MONTHS 2–3)
   - udica-generated per-service policies
   - Start in permissive mode, log violations, refine
   - Graduated enforcing mode for gateway container
   - Upstream intention: Fedora SELinux SIG submission
   - Files: implementation/phase1/selinux/

By month 3, you will also build:

D. SECURITY AGENT — TRUST LEVEL ENFORCEMENT
   - TRUST_0: silent, read-only, auto-approved
   - TRUST_1: notification + inline undo capability
   - TRUST_2: confirm/deny user prompt
   - TRUST_3: modal with diff preview
   - Escalation rules: scope creep → +1 level; unexpected network → TRUST_3
   - Revocation: 3 violations in 60s → suspend agent
   - Files: implementation/phase1/security_agent/

9. READING RESOURCES
============================================================

ESSENTIAL (read in Month 1):

1. OAuth 2.0 Simplified — Aaron Parecki
   https://www.oauth.com/ (free online book)
   Focus: Chapters 1–6 (authorization code, client credentials, tokens)

2. Linux Secret Service API Specification
   https://specifications.freedesktop.org/secret-service/latest/
   Focus: Sections on storing/retrieving secrets, session management

3. Python keyring library documentation
   https://pypi.org/project/keyring/
   Focus: get_password, set_password, delete_password

4. Podman Rootless Tutorial
   https://github.com/containers/podman/blob/main/docs/tutorials/rootless_tutorial.md
   Focus: User namespaces, capabilities, SELinux with :Z

5. Podman Quadlet Documentation
   https://docs.podman.io/en/latest/markdown/podman-systemd.unit.5.html
   Focus: .container file format, security directives

RECOMMENDED (read in Month 2):

6. SELinux Coloring Book — Red Hat
   Free PDF: search "SELinux Coloring Book Red Hat"
   Best beginner resource for SELinux. Read the whole thing.

7. Fedora SELinux User Guide
   https://docs.fedoraproject.org/en-US/quick-docs/selinux-getting-started/
   Focus: permissive vs enforcing, ausearch, audit2allow

8. udica: SELinux Policy Generator for Containers
   https://github.com/containers/udica
   Focus: HOWTO section, generate and inspect policies

9. Linux Capabilities man page
   man 7 capabilities
   Focus: CAP_SYS_ADMIN, CAP_NET_RAW, and what each container needs

RECOMMENDED (read in Month 3):

10. STRIDE Threat Modeling — Microsoft
    https://learn.microsoft.com/en-us/azure/security/develop/threat-modeling-tool-threats
    Focus: Each STRIDE category, example threats

11. OWASP API Security Top 10 (2023)
    https://owasp.org/API-Security/
    Focus: Broken authentication, excessive data exposure, lack of resources

12. Object-Capability Model
    https://en.wikipedia.org/wiki/Object-capability_model
    Focus: How capabilities differ from ACLs, revocation
